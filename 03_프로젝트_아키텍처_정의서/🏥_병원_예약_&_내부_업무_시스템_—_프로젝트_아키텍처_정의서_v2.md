## 2. 기술 스택

| 구분 | 기술 | 비고 |
| --- | --- | --- |
| **Language** | Java 17+ | LTS |
| **Framework** | Spring Boot 3.x | Spring MVC, Spring Security, Spring Data JPA |
| **Template Engine** | Mustache | SSR (Server-Side Rendering) |
| **Database** | MySQL 8.x | 운영 DB |
| **ORM** | Spring Data JPA (Hibernate) | Entity 매핑 |
| **세션 저장소** | Redis (Spring Session Data Redis) | PBL 필수 — spring-session-data-redis |
| **인증/인가** | Spring Security | Session 기반, ROLE별 접근 제어 |
| **빌드** | Gradle | build.gradle |
| **HTTP Client** | Spring WebClient | Claude API 호출용 (비동기 논블로킹) |
| **테스트** | JUnit5 + MockMvc + @DataJpaTest | PBL 권장 |

## 4. 담당자별 소유 영역 전체 지도

### 4.1 책임개발자 (Tech Lead)

> **철학:** 뼈대를 세운다. 개발자들이 채워 넣을 틀을 만든다.
>

| 영역 | 파일 / 클래스 | 완료 목표 |
| --- | --- | --- |
| **인프라·설정** | SecurityConfig, WebMvcConfig, application.properties | W1 중반 |
| **공통 레이아웃** | LayoutModelInterceptor — pageTitle·loginName·isRole·showChatbot 자동 주입 | W1 중반 |
| **Entity 전체** | Patient, Staff, Doctor, Department, Reservation, TreatmentRecord, Item, HospitalRule, LlmRecommendation, ChatbotHistory | W1 전반 |
| **예약 핵심 로직** | ReservationValidationService, SlotService (중복 방지 트랜잭션) | W1 후반 |
| **예약번호 발급** | ReservationNumberGenerator.java | W1 후반 |
| **예외 처리** | GlobalExceptionHandler, CustomException | W1 후반 |
| **LLM 서비스** | LlmService (Claude API 호출·파싱·폴백) | W3 말 착수 → W4 초 완료 |
| **LLM DTO** | SymptomRequest/Response, ChatbotRequest/Response | W3 말 확정 후 팀 공유 |
| **배포·보안 점검** | .gitignore, 환경변수, Linux 서버 배포 스크립트 | W4 후반 |

**AI 활용:** Entity 초안 생성, SecurityConfig URL 패턴 검토, LLM 프롬프트 초안

### 4.2 개발자 A — 외부 예약 흐름

> **담당 URL 영역:** `/` · `/reservation/**`**담당 화면:** 00 (비회원 메인) · 01~04 (예약 흐름 전체)

>

| 영역 | 파일 / 클래스 | 완료 목표 |
| --- | --- | --- |

| ReservationController | `GET /reservation`, `GET/POST /reservation/symptom`, `GET/POST /reservation/direct`, `GET /reservation/complete` | W2 |
| ReservationService | 예약 생성 (`POST /reservation/create`) — **SlotService 호출로 중복 방지** | W2 |

| ReservationRepository | Reservation 저장·조회 | W2 |
| Mustache 템플릿 | `home/index.mustache`, `reservation/*.mustache` | W1~W2 |
| LLM 추천 UI 연결 | 증상 입력 폼 → `POST /llm/symptom/analyze` 호출 + 결과 렌더링 | W4 (LlmService 완성 후) |

**AI 활용:** 예약 폼 Mustache 템플릿, 의사 동적 조회 JS, 슬롯 선택 JS, LLM 추천 결과 렌더링 HTML

## 5. 공유 레이어 — 책임개발자 단독 소유

개발자들이 의존하지만 **절대 수정 불가**한 공유 코드.
수정이 필요하면 반드시 책임개발자에게 요청.

### 5.1 SlotService (예약 중복 방지)

```java
// com.hospital.common.service.SlotService.java  ◆ 책임개발자 소유
public interface SlotService {
    List<String> getAvailableSlots(Long doctorId, LocalDate date);
    void validateAndLock(Long doctorId, LocalDate date, String timeSlot); // @Transactional + UNIQUE 검증
}
```

> 개발자 A의 `ReservationService`와 개발자 B의 `WalkinService`가 **반드시 이 인터페이스를 호출**해야 함. 직접 Reservation 저장 시 중복 허용 버그 발생.
>

### 5.2 LayoutModelInterceptor (공통 레이아웃 모델 자동 주입)

```java
// com.hospital.common.interceptor.LayoutModelInterceptor.java  ◆ 책임개발자 소유
// 모든 Controller에서 Model.addAttribute 불필요
// Interceptor가 자동으로 주입하는 Model 키:
//   pageTitle, loginName, loginRole
//   isStaff, isDoctor, isNurse, isAdmin
//   showChatbot, currentMenu
```

> 개발자들은 Controller에서 `pageTitle`과 `currentMenu`만 설정하면 됨.
나머지 ROLE 관련 모델은 Interceptor가 세션에서 자동 추출.
>

### 5.3 Entity 클래스 (변경 금지)

모든 `domain/*.java` Entity는 책임개발자만 수정.
개발자가 필드 추가나 컬럼 수정이 필요하면 → 책임개발자에게 GitHub Issue 등록.

## 9. CSRF 토큰 + AJAX 처리

### 원칙

Spring Security CSRF 보호가 기본 활성화 상태. SSR 폼은 자동 처리되지만 AJAX 요청은 수동 전달 필요.

### 방법 1: Mustache 메타 태그 방식 (권장)

공통 레이아웃 `<head>`에 CSRF 메타 태그 삽입:

```html
<meta name="_csrf" content="{{_csrf.token}}"/>
<meta name="_csrf_header" content="{{_csrf.headerName}}"/>
```

AJAX 공통 유틸리티 (`static/js/common.js`):

```javascript
function fetchWithCsrf(url, options = {}) {
    const token = document.querySelector('meta[name="_csrf"]').content;
    const header = document.querySelector('meta[name="_csrf_header"]').content;
    options.headers = { ...options.headers, [header]: token, 'Content-Type': 'application/json' };
    return fetch(url, options);
}
```

### 소유권

공통 레이아웃의 메타 태그와 `common.js`는 책임개발자 소유. 각 개발자는 `fetchWithCsrf()`를 호출하여 사용.

## 10. 페이징 구현 패턴 (PBL 필수)

## 12. AI 보조 역할 범위

> AI는 **코드 초안·템플릿·디버깅 보조** 역할. 설계 결정·코드 리뷰는 책임개발자.
>

| 담당자 | AI 활용 가능 범위 | AI에게 요청하면 안 되는 것 |
| --- | --- | --- |
| **책임개발자** | SecurityConfig URL 패턴 검토, LLM 프롬프트 초안, Entity 초안, 배포 스크립트 | 아키텍처 결정, 트랜잭션 경계 설계 |
| **개발자 A** | 예약 폼 Mustache 초안, 슬롯 동적 조회 JS, LLM 추천 결과 렌더링 | SlotService 로직 직접 구현 요청, Entity 수정 요청 |
| **개발자 B** | 대시보드 템플릿, 접수 처리 폼, 챗봇 오버레이 JS, 상태 배지 CSS | TreatmentService 상태 전이 로직 임의 구현 |
| **개발자 C** | CRUD 폼 Mustache 일괄 생성, 페이징 컴포넌트, 통계 집계 JPQL | HospitalRule과 LLM 연결 로직 직접 구현 |

### AI 사용 표준 절차

```
1. 구현 전 → AI에게 "이 기능의 담당자가 나(개발자X)가 맞는지" 확인
2. 코드 초안 생성 → AI 활용 (빠른 시작)
3. 타 담당자 영역 파일 수정 필요 시 → AI에게 요청하지 말고 책임개발자에게 에스컬레이션
4. PR 전 → AI에게 "이 PR에 타 담당자 파일이 포함되어 있는지" 검토 요청
5. 30분 이상 블로킹 → AI 디버깅 보조 요청 + 동시에 책임개발자에게 DM
```

## 13. 인터페이스 계약 — 담당자 간 연결 지점

> 담당자 간 경계에서 서로 호출하는 지점을 명시. 이 계약을 어기면 충돌 발생.
>

### 13.1 책임개발자 → 개발자 A (SlotService)

```java
// ◆ 책임개발자가 W1 후반까지 완성해 develop에 머지
// ▲ 개발자 A는 이 인터페이스만 호출하면 됨 (구현 불필요)

// com.hospital.common.service.SlotService
List<String> getAvailableSlots(Long doctorId, LocalDate date);
// → 사용 가능한 시간 슬롯 반환 (이미 예약된 슬롯 제외)

void validateAndLock(Long doctorId, LocalDate date, String timeSlot);
// → 중복이면 ReservationConflictException 발생 (자동 롤백)
// → 정상이면 예약 저장까지 같은 트랜잭션에서 처리
```

**약속:** 책임개발자는 W1 말까지 `SlotService` 구현 완성 + `develop` 브랜치 머지.
개발자 A는 W2 시작 시 pull 후 의존.

---

### 13.2 책임개발자 → 개발자 A·B (LayoutModelInterceptor)

```java
// ◆ 책임개발자가 W1 중반까지 완성
// ▲▲ 개발자들은 Controller에서 아래 두 가지만 설정

// 모든 내부 화면 Controller에서:
model.addAttribute("pageTitle", "접수 목록");  // 화면 이름
```

## 15. 충돌 방지 규칙 (Anti-Collision Rules)

### 15.1 파일 소유권 원칙

```
규칙 1. "기능 단위 수직 소유"
  한 기능의 Controller + Service + Repository + Mustache 템플릿을
  동일한 담당자가 소유한다.
  → 다른 담당자의 Service를 수정하거나 메서드를 추가하려면
    반드시 책임개발자 승인 후 해당 담당자가 작업.

규칙 2. "공유 파일 터치 금지"
  Entity 클래스, SecurityConfig, LayoutModelInterceptor, SlotService는
  책임개발자만 수정. 개발자는 수정 필요 시 GitHub Issue 등록.

규칙 3. "Repository 읽기·쓰기 경계"
  자신의 기능 담당이 아닌 Repository를 직접 호출할 때는 읽기(findById 등)만 허용.
  상태 변경(save, delete)은 해당 Service를 통해서만.

규칙 4. "상태 전이 단방향 소유"
  RESERVED → RECEIVED:  ReceptionService (개발자 B)만 처리
  RECEIVED → COMPLETED:  TreatmentService (개발자 B)만 처리
  RESERVED → CANCELLED:  AdminReservationService (개발자 C)만 처리
  RECEIVED 직행:          WalkinService (개발자 B)만 처리
  → 같은 상태 전이를 두 곳에서 하면 버그 원인

규칙 5. "Mustache 템플릿 디렉터리 소유"
  templates/home/**       → 개발자 A
  templates/reservation/** → 개발자 A
  templates/staff/**      → 개발자 B
  templates/doctor/**     → 개발자 B
  templates/nurse/**      → 개발자 B
  templates/admin/**      → 개발자 C
  templates/common/**     → 책임개발자 (레이아웃·헤더·사이드바·푸터·챗봇 파셜)
  templates/layouts/**    → 책임개발자
```

### 15.2 충돌 발생 시 해결 절차

```
1단계: 발견 즉시 작업 중단
2단계: GitHub Issue에 "충돌: [파일명] [담당자1] vs [담당자2]" 등록
3단계: 책임개발자가 소유권 판단 및 해결 방향 결정 (30분 내)
4단계: 소유자가 수정, 비소유자는 rebase로 최신화
5단계: PR 재요청 및 책임개발자 최종 리뷰
```
