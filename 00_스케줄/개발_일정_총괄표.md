## 5. 의존성 체인 (Critical Path)

```
W1 책임개발자(김민구): Entity + SecurityConfig + SlotService + LayoutModelInterceptor 완성
  │
  ├──→ W2 개발자 A(강태오): SlotService 의존 → 예약 폼 구현
  ├──→ W2 개발자 B(조유지): SlotService 의존 → 방문 접수 구현
  └──→ W2 개발자 C(강상민): Entity 의존 → 관리자 CRUD 구현
        │
        └──→ W3 전체: 각자 영역 확장 개발
              │
└──→ W3 후반 책임개발자(김민구): LlmService 선행 착수
      │
      ├──→ W4 개발자 A(강태오): SymptomResponse DTO 의존 → 증상 UI 연결
      ├──→ W4 개발자 B(조유지): ChatbotResponse DTO 의존 → 챗봇 UI 연결
      └──→ W4 책임개발자(김민구): 통합 테스트 + 배포
```

## 6. 리스크 대응 타임라인

| 시점 | 체크 항목 | 정상 경로 | 대응 플랜 |
|------|-----------|-----------|-----------|
| W1 금 | Entity·Security 완성 여부 | 개발자 W2 병렬 개발 시작 | 주말 긴급 완성, 개발자 UI만 선행 |
| W2 금 | 예약·접수 E2E 동작 여부 | W3 진료·관리자 개발 시작 | 예약 흐름 우선 마무리, W3 전반에 접수 보완 |
| W3 수 | LlmService 기본 동작 여부 | W3 금 DTO 공유 미팅 | 개발자 A(강태오): 직접 선택 예약 마무리, 개발자 B(조유지): 더미 챗봇 UI |
| W3 금 | LlmService DTO 확정 여부 | W4 UI 연결 시작 | 최악 시 챗봇 제외, 증상 추천만 배포 |
| W4 수 | LLM UI 연결 완료 여부 | 통합 테스트 진행 | 미완성 기능 Scope 축소 결정 |
| W4 금 | 전체 MVP 배포 가능 여부 | 배포 완료 | 핫픽스 브랜치로 긴급 수정 |

## 7. 테스트 일정 (주차별)

| 주차 | 테스트 활동 | 담당 |
|------|------------|------|
| **W1** | 테스트 환경 세팅, `SecurityConfig` 테스트, 테스트 유틸 클래스 작성 | 책임개발자(김민구) |
| **W2** | Service/Repository 단위 테스트 병행 (예약, 접수, 환자) | 각 기능 담당자 |
| **W3** | Controller MockMvc 테스트, LLM Mock 테스트, SlotService 테스트 | 각 기능 담당자 + 책임개발자(김민구) |
| **W4** | 통합 테스트 (전체 흐름), 회귀 테스트, JaCoCo 커버리지 확인 | 책임개발자(김민구) 주도, 전원 참여 |
