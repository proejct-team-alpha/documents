## 2. 서비스 시작 · 중지 · 재시작

### 2.1 전체 서비스

```bash
# 전체 기동 (백그라운드)
docker compose up -d

# 전체 기동 (빌드 포함)
docker compose up -d --build

# 전체 중지 (데이터 보존)
docker compose down

# 전체 중지 + 볼륨 삭제 (⚠️ 데이터 삭제됨)
docker compose down -v

# 전체 재시작
docker compose restart

# 상태 확인
docker compose ps
```

### 2.2 개별 서비스

```bash
# 특정 서비스 재시작
docker compose restart spring-app
docker compose restart python-llm
docker compose restart nginx
docker compose restart mysql
docker compose restart redis
docker compose restart chromadb

# 특정 서비스만 재빌드 후 기동
docker compose up -d --build spring-app
docker compose up -d --build python-llm

# 특정 서비스 중지
docker compose stop spring-app

# 특정 서비스 로그
docker compose logs -f spring-app
```

### 2.3 Graceful Shutdown

Spring Boot는 `server.shutdown=graceful` 설정이 적용되어 있어 종료 시 진행 중인 요청을 최대 30초간 완료 대기합니다.

```properties
# application-prod.properties
server.shutdown=graceful
spring.lifecycle.timeout-per-shutdown-phase=30s
```

```bash
# Graceful 종료 (SIGTERM → 30초 대기 → 강제 종료)
docker compose stop spring-app

# 강제 종료가 필요한 경우
docker compose kill spring-app
```

### 3.3 자동 헬스 체크 스크립트

```bash
#!/bin/bash
# healthcheck.sh — 크론으로 주기 실행 권장

ALERT_CMD="echo"  # 실제 환경에서 슬랙/이메일 알림으로 교체

# Spring Boot
HTTP_CODE=$(curl -k -s -o /dev/null -w "%{http_code}" https://localhost/login 2>/dev/null)
if [ "$HTTP_CODE" != "200" ]; then
    $ALERT_CMD "[HMS ALERT] Spring Boot 응답 이상: HTTP $HTTP_CODE"
fi

# MySQL
docker exec hms-db mysqladmin ping -h localhost 2>/dev/null | grep -q "alive"
if [ $? -ne 0 ]; then
    $ALERT_CMD "[HMS ALERT] MySQL 응답 없음"
fi

# Redis
PONG=$(docker exec hms-redis redis-cli ping 2>/dev/null)
if [ "$PONG" != "PONG" ]; then
    $ALERT_CMD "[HMS ALERT] Redis 응답 없음"
fi

# Python LLM
LLM_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8000/docs 2>/dev/null)
if [ "$LLM_CODE" != "200" ]; then
    $ALERT_CMD "[HMS ALERT] Python LLM 서버 응답 이상: HTTP $LLM_CODE"
fi

echo "[$(date)] 헬스 체크 완료"
```

```bash
# 크론 등록 (5분마다 실행)
crontab -e
# */5 * * * * /opt/hms/scripts/healthcheck.sh >> /opt/hms/logs/healthcheck.log 2>&1
```

### 3.4 LLM 서비스 메트릭

Python LLM 서버는 내장 메트릭을 수집합니다.

| 메트릭            | 설명                |
| ----------------- | ------------------- |
| `total_requests`  | 총 추론 요청 수     |
| `success_count`   | 성공 횟수           |
| `error_count`     | 실패 횟수           |
| `avg_latency_ms`  | 평균 응답 시간 (ms) |
| `success_rate`    | 성공률 (0~1)        |
| `vector_hit_rate` | 벡터 검색 적중률    |

### 3.5 Circuit Breaker 상태

Python LLM 서버에 구현된 Circuit Breaker 패턴:

| 상태               | 설명                                                     |
| ------------------ | -------------------------------------------------------- |
| `CLOSED` (정상)    | 요청 정상 처리, 실패 카운트 중                           |
| `OPEN` (차단)      | 연속 5회 실패 시 전환, 30초간 요청 즉시 차단             |
| `HALF_OPEN` (시험) | 30초 경과 후 단일 프로브 요청 허용 → 성공 시 CLOSED 복귀 |

> Circuit Breaker가 OPEN 상태이면 LLM 요청이 즉시 실패하여 사용자 대기 시간을 줄입니다.

## 4. 로그 관리

### 4.1 로그 위치

| 서비스               | 로그 위치                                  | 접근 방법                                                               |
| -------------------- | ------------------------------------------ | ----------------------------------------------------------------------- |
| **Spring Boot**      | 컨테이너 내부: `/opt/hms/logs/hms-app.log` | `docker compose logs spring-app` 또는 `tail -f logs/spring/hms-app.log` |
| **Spring Boot 에러** | `/opt/hms/logs/hms-error.log`              | `tail -f logs/spring/hms-error.log`                                     |
| **Python LLM**       | 컨테이너 stdout                            | `docker compose logs -f python-llm`                                     |
| **Nginx**            | 컨테이너 stdout                            | `docker compose logs -f nginx`                                          |
| **MySQL**            | 컨테이너 내부                              | `docker compose logs -f mysql`                                          |

### 4.2 로그 설정 (Logback)

HMS는 `logback-spring.xml`로 프로필별 로깅을 관리합니다.

#### prod 프로필 로그 구성

| Appender           | 대상            | 설명                                    |
| ------------------ | --------------- | --------------------------------------- |
| `CONSOLE_PLAIN`    | stdout          | Plain 텍스트 콘솔 출력                  |
| `ASYNC_FILE`       | `hms-app.log`   | 롤링 파일 (50MB × 90일, 최대 5GB)       |
| `ASYNC_ERROR_FILE` | `hms-error.log` | ERROR 레벨 전용 (50MB × 90일, 최대 2GB) |

#### 로그 레벨

| 로거                  | prod 레벨 |
| --------------------- | --------- |
| `root`                | `WARN`    |
| `com.smartclinic.hms` | `INFO`    |

### 4.3 로그 롤링 정책

```
logs/
├── hms-app.log                     ← 현재 로그
├── hms-error.log                   ← 현재 에러 로그
└── archive/
    ├── hms-app.2026-03-24.0.log.gz ← 일별 아카이브 (gzip)
    ├── hms-app.2026-03-24.1.log.gz ← 50MB 초과 시 분할
    └── hms-error.2026-03-24.0.log.gz
```

- **파일 크기 제한:** 50MB/파일
- **보관 기간:** 90일
- **총 용량 제한:** app 5GB, error 2GB
- 비동기 Appender(큐 사이즈 1024) 사용으로 I/O 병목 방지

### 4.4 로그 검색

```bash
# 에러만 필터링 (최근 50건)
grep -i "error\|exception" logs/spring/hms-app.log | tail -50

# 특정 시간대 로그
grep "2026-03-25 14:" logs/spring/hms-app.log

# 특정 요청 추적 (예약번호로)
grep "RES-" logs/spring/hms-app.log

# Python LLM 에러
docker compose logs python-llm 2>&1 | grep -i "error\|exception\|traceback"

# Nginx 4xx/5xx 에러
docker compose logs nginx 2>&1 | grep -E " (4[0-9]{2}|5[0-9]{2}) "
```

### 4.5 로그 정리

```bash
# 30일 이상 된 아카이브 수동 삭제 (Logback이 자동 관리하지만 디스크 긴급 시)
find logs/spring/archive/ -name "*.log.gz" -mtime +30 -delete
```

## 5. 데이터베이스 운영

### 5.1 MySQL 접속

```bash
# 컨테이너 내부에서 접속
docker exec -it hms-db mysql -u hms_admin -phms_password hms_db

# 호스트에서 직접 접속
```

### 5.2 주요 테이블

| 테이블               | 설명                    | 데이터 규모 예상            |
| -------------------- | ----------------------- | --------------------------- |
| `department`         | 진료과                  | 수십 건                     |
| `staff`              | 직원 (로그인 계정)      | 수십~수백 건                |
| `doctor`             | 의사 (진료 스케줄 포함) | 수십 건                     |
| `patient`            | 환자                    | 수천~수만 건                |
| `reservation`        | 예약                    | 지속 증가 (일 수십~수백 건) |
| `treatment_record`   | 진료 기록               | reservation과 1:1           |
| `hospital_rule`      | 병원 규칙               | 수십~수백 건                |
| `item`               | 물품                    | 수백 건                     |
| `chatbot_history`    | 챗봇 대화 이력          | 지속 증가                   |
| `llm_recommendation` | LLM 추천 기록           | 지속 증가                   |

### 5.3 일상 점검 쿼리

```sql
-- 오늘 예약 현황 요약
SELECT status, COUNT(*) as cnt
FROM reservation
WHERE reservation_date = CURDATE()
GROUP BY status;

-- 활성 직원 수 (역할별)
SELECT role, COUNT(*) as cnt
FROM staff
WHERE is_active = true
GROUP BY role;

-- 테이블별 레코드 수
SELECT TABLE_NAME, TABLE_ROWS
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'hms_db'
ORDER BY TABLE_ROWS DESC;

-- 슬로우 쿼리 확인 (MySQL 8.0)
SHOW GLOBAL STATUS LIKE 'Slow_queries';

-- 커넥션 풀 상태
SHOW STATUS LIKE 'Threads_%';
SHOW STATUS LIKE 'Max_used_connections';
```

### 5.4 HikariCP 커넥션 풀 설정

```properties
# application-prod.properties
spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=3000
```

| 파라미터             | 값     | 설명                  |
| -------------------- | ------ | --------------------- |
| `maximum-pool-size`  | 20     | 최대 커넥션 수        |
| `minimum-idle`       | 5      | 유휴 커넥션 최소 수   |
| `connection-timeout` | 3000ms | 커넥션 획득 대기 시간 |

> **경고:** `maximum-pool-size`를 MySQL의 `max_connections` 이하로 유지해야 합니다. Python LLM 서버도 MySQL에 접속하므로 총 커넥션 수를 고려하세요.

### 5.5 JPA DDL 정책

| 환경 | ddl-auto      | 설명                       |
| ---- | ------------- | -------------------------- |
| dev  | `create-drop` | 기동 시 생성, 종료 시 삭제 |
| prod | `update`      | 스키마 변경분만 자동 적용  |

> **운영 안정화 후:** `update` → `validate` 또는 `none`으로 변경하고 DDL은 별도 마이그레이션 스크립트로 관리할 것을 권장합니다.

## 6. Redis 운영

### 6.1 역할

| 용도          | 설명                                                                        |
| ------------- | --------------------------------------------------------------------------- |
| **캐시**      | 진료과별 의사 목록 (1시간 TTL), 의사 스케줄 (5분 TTL), 기본 캐시 (10분 TTL) |
| **세션 저장** | Spring Security 세션 (prod 환경)                                            |

### 6.2 캐시 설정

```java
// CacheConfig.java — Redis 캐시 매니저
"doctors-by-department"  → TTL 1시간
"doctors-with-schedule"  → TTL 5분
기본                     → TTL 10분
```

### 6.3 일상 점검

```bash
# Redis 접속
docker exec -it hms-redis redis-cli

# 메모리 사용량
docker exec hms-redis redis-cli info memory | grep used_memory_human

# 키 수
docker exec hms-redis redis-cli dbsize

# 캐시 키 패턴 확인
docker exec hms-redis redis-cli keys "doctors*"

# 세션 키 확인
docker exec hms-redis redis-cli keys "spring:session*"

# 슬로우 로그
docker exec hms-redis redis-cli slowlog get 10
```

### 6.4 캐시 초기화

```bash
# 전체 캐시 초기화 (세션도 삭제됨 — 모든 사용자 재로그인 필요)
docker exec hms-redis redis-cli FLUSHALL

# 특정 캐시만 삭제
docker exec hms-redis redis-cli DEL "doctors-by-department::1"
```

> **주의:** `FLUSHALL`은 세션 데이터도 삭제하므로 모든 로그인 사용자가 로그아웃됩니다. 운영 중에는 특정 키만 삭제하세요.

### 7.3 ChromaDB 벡터 데이터 관리


## 8. 보안 운영

### 8.1 Rate Limiting

IP 기반 토큰 버킷 방식의 Rate Limiting이 적용되어 있습니다.

| 엔드포인트        | 제한     | 윈도우 | 목적            |
| ----------------- | -------- | ------ | --------------- |
| `POST /login`     | 10회/분  | 60초   | 브루트포스 방지 |
| `/llm/symptom/**` | 20회/분  | 60초   | API 비용 제어   |
| 기타              | 100회/분 | 60초   | 일반 남용 방지  |

초과 시 `429 Too Many Requests` + JSON 에러 응답 반환.

> **설정:** 정적 리소스(`/css/`, `/js/`, `/images/`, `/favicon.ico`)는 Rate Limit에서 제외됩니다.

### 8.2 CSRF 보호

| 경로                  | CSRF       | 이유                  |
| --------------------- | ---------- | --------------------- |
| `/llm/symptom/**`     | 비활성화   | 비회원 AJAX JSON 전용 |
| `/llm/medical/**`     | 비활성화   | 비회원 AJAX JSON 전용 |
| `/llm/chatbot/**`     | 비활성화   | 내부 AJAX JSON 전용   |
| `/llm/reservation/**` | 비활성화   | 비회원 AJAX JSON 전용 |
| 나머지 모든 경로      | **활성화** | SSR 폼 보호           |

> JS fetch 호출 시 CSRF 토큰을 `X-CSRF-TOKEN` 헤더로 전송해야 합니다.

### 8.3 Security Headers

| 헤더                          | 설정                                                                |
| ----------------------------- | ------------------------------------------------------------------- |
| **Content-Security-Policy**   | `default-src 'self'` 기반, 외부 CDN(jsdelivr, 카카오 우편번호) 허용 |
| **Strict-Transport-Security** | `max-age=31536000; includeSubDomains`                               |
| **Referrer-Policy**           | `strict-origin-when-cross-origin`                                   |
| **Permissions-Policy**        | `camera=(), microphone=(), geolocation=(), payment=()`              |

### 8.4 세션 관리

| 설정          | 값                                   | 설명                        |
| ------------- | ------------------------------------ | --------------------------- |
| 세션 타임아웃 | 30분 (prod)                          | 비활동 시 자동 만료         |
| 동시 세션     | 1개                                  | 새 로그인 시 기존 세션 만료 |
| 쿠키 속성     | `HttpOnly`, `Secure`, `SameSite=Lax` | XSS·CSRF 방어               |
| 로그아웃      | 세션 무효화 + JSESSIONID 삭제        | 완전 정리                   |

## 9. 백업 및 복구

### 9.1 MySQL 백업

#### 일일 자동 백업 스크립트

```bash
#!/bin/bash
# mysql-backup.sh

BACKUP_DIR="/opt/hms/backup/mysql"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
RETAIN_DAYS=30

mkdir -p "$BACKUP_DIR"

# mysqldump 실행 (컨테이너 내부)
docker exec hms-db mysqldump \
  -u root -prootpassword \
  --single-transaction \
  --routines \
  --triggers \
  hms_db | gzip > "$BACKUP_DIR/hms_db_${TIMESTAMP}.sql.gz"

if [ $? -eq 0 ]; then
    echo "[$(date)] MySQL 백업 성공: hms_db_${TIMESTAMP}.sql.gz"
else
    echo "[$(date)] MySQL 백업 실패!" >&2
    exit 1
fi

# 오래된 백업 삭제
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +${RETAIN_DAYS} -delete
echo "[$(date)] ${RETAIN_DAYS}일 이상 된 백업 삭제 완료"
```

```bash
# 크론 등록 (매일 새벽 2시)
crontab -e
# 0 2 * * * /opt/hms/scripts/mysql-backup.sh >> /opt/hms/logs/backup.log 2>&1
```

#### 수동 백업

```bash
# 전체 백업
docker exec hms-db mysqldump -u root -prootpassword --single-transaction hms_db | gzip > backup_$(date +%Y%m%d).sql.gz

# 특정 테이블만 백업
docker exec hms-db mysqldump -u root -prootpassword --single-transaction hms_db reservation treatment_record | gzip > backup_reservation_$(date +%Y%m%d).sql.gz
```

```bash
# 5. 벡터 데이터 손상 시 재인덱싱
docker exec -it hms-python python index_medical_data.py
docker exec -it hms-python python index_rule_data.py
```

#### 시나리오 5: Nginx 다운 (P1)

```bash
# 1. 상태 확인
docker compose ps nginx

# 2. 설정 검증
docker exec hms-nginx nginx -t

# 3. SSL 인증서 확인
openssl x509 -enddate -noout -in nginx/ssl/server.crt

# 4. 재시작
docker compose restart nginx

# 5. 설정 오류 시 기본 설정으로 복원
git checkout nginx/default.conf
docker compose restart nginx
```

#### 시나리오 6: 디스크 부족 (P2)

```bash
# 1. 디스크 사용량 확인
df -h
du -sh /opt/hms/backup/*
du -sh logs/spring/*

# 2. Docker 리소스 정리
docker system prune -f
docker volume prune -f  # ⚠️ 미사용 볼륨만 삭제됨

# 3. 오래된 백업 삭제
find /opt/hms/backup -name "*.sql.gz" -mtime +7 -delete

# 4. 오래된 로그 삭제
find logs/spring/archive/ -name "*.log.gz" -mtime +30 -delete
```
