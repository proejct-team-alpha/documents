> **분석 일자:** 2026-03-02
> **조치 완료:** 2026-03-02

## CRITICAL (6건)

### C-01. 내 정보관리(MyPage) 화면 — 스토리보드에만 존재 ✅ 조치 완료

> **조치:** API 명세서 v5에 8개 엔드포인트 추가, 화면 기능 정의서 v5에 4개 화면(33~36) 추가, 화면 흐름 시퀀스 v5에 4개 역할별 흐름 추가

스토리보드에 4개 내 정보관리 화면이 정의되어 있으나, **API 명세서·화면 기능 정의서·화면 흐름 시퀀스 다이어그램 어디에도 존재하지 않음**.

| 화면 번호 | 화면명 | URL (GET) | POST URL |
|-----------|--------|-----------|----------|
| 40 | STAFF 내 정보관리 | `/staff/mypage` | `/staff/mypage/update` |
| 20 | DOCTOR 내 정보관리 | `/doctor/mypage` | `/doctor/mypage/update` |
| 21 | NURSE 내 정보관리 | `/nurse/mypage` | `/nurse/mypage/update` |
| 31 | ADMIN 내 정보관리 | `/admin/mypage` | (미명시) |

**영향:** API 8개(GET 4 + POST 4), 화면 4개, 시퀀스 4개 흐름 누락

---

### C-02. 화면 번호 충돌 — 화면 19, 20, 21, 31 ⚠️ 보류

> **보류 사유:** 기존 정의서 번호 체계(19=접수목록, 20=환자목록, 21=환자상세, 31=규칙등록) 유지. 내 정보관리는 33~36 신규 번호 배정. 스토리보드와의 번호 차이는 구현 시 매핑으로 해결.

스토리보드와 화면 기능 정의서가 동일 번호에 **완전히 다른 화면**을 배정.

| 화면 번호 | 스토리보드 | 화면 기능 정의서 v4 |
|-----------|-----------|-------------------|
| **19** | 예약 상세·취소 (`/admin/reservation/detail`) | 전체 접수 목록 (`/admin/reception/list`) |
| **20** | DOCTOR 내 정보관리 (`/doctor/mypage`) | 환자 목록 (`/admin/patient/list`) |
| **21** | NURSE 내 정보관리 (`/nurse/mypage`) | 환자 상세 (`/admin/patient/detail`) |
| **31** | 규칙 등록 **AND** 물품담당 내 정보관리 (자체 충돌) | 병원 규칙 등록 (`/admin/rule/new`) |

---

### C-03. 화면 01(예약 방식 선택) — 스토리보드에서 누락 ⚠️ 보류

> **보류 사유:** 화면 01은 정의서 v5에 그대로 유지. 스토리보드가 직행하는 것은 UI 단축 표현으로 판단. 구현 시 화면 01 유지.

화면 기능 정의서·화면 흐름에 존재하는 `GET /reservation` (예약 방식 선택) 화면이 스토리보드에 없음.
스토리보드는 메인(화면 00) → 증상 입력(화면 02)으로 직행하여 중간 단계를 건너뜀.

---

### C-04. 관리자 환자 관리 화면(20, 21) — 스토리보드에서 누락 ⚠️ 보류

> **보류 사유:** 정의서 v5에 기존 화면 20, 21 유지. 스토리보드에 누락된 것은 PPTX 미수정 방침에 따라 그대로 둠.

화면 기능 정의서 v4에 정의된 관리자 환자 관리 화면 2개가 스토리보드에 없음.

| 정의서 화면 | URL | 설명 |
|------------|-----|------|
| 화면 20 | `GET /admin/patient/list` | 환자 목록 |
| 화면 21 | `GET /admin/patient/detail` | 환자 상세 |

---

### C-05. TREATMENT_RECORD 필드 구조 불일치 ✅ 조치 완료

> **조치:** ERD v4에서 `diagnosis_note` → `diagnosis`(TEXT NOT NULL) + `prescription`(TEXT NOT NULL) + `remark`(TEXT NULL) 3개 필드로 분리

| 항목 | 스토리보드 (화면 13) | ERD v3 |
|------|---------------------|--------|
| 필드 수 | **3개**: 진단명\*, 처방\*, 비고(선택) | **1개**: `diagnosis_note` (TEXT, NULL) |
| NOT NULL | 진단명·처방 = 필수 | `diagnosis_note` = NULL 허용 |

스토리보드는 3개 분리 필드를 기대하나, ERD에는 단일 TEXT 컬럼만 존재.

---

### C-06. 사원번호(employee_number) 컬럼 — ERD에 없음 ✅ 조치 완료

> **조치:** ERD v4에 `STAFF.employee_number` VARCHAR(20) NOT NULL UNIQUE 추가. 형식: `{역할코드}-{YYYYMMDD}` (예: D-20190305)

스토리보드의 내 정보관리 화면에서 사원번호를 `D-20190305`, `S-20230401`, `N-20210812`, `I-20220601` 형식으로 표시하나, ERD STAFF 테이블에 해당 컬럼이 존재하지 않음. `username`과도 형식이 다름.

## HIGH (8건)

### H-01. API 명세서에 내 정보관리 엔드포인트 없음 ✅ 조치 완료

### H-02. 필터 파라미터 불일치 — API 명세서 vs 스토리보드 ✅ 조치 완료

> **조치:** API 명세서 v5에서 7개 목록 엔드포인트에 스토리보드 기준 필터 파라미터 추가 (status, departmentId, keyword, page 등)

7개 화면에서 스토리보드가 기대하는 필터가 API 명세서에 미정의.

| 화면 | 스토리보드 필터 | API 명세서 필터 | 누락 |
|------|---------------|----------------|------|
| 07 접수 목록 | status, departmentId, keyword, page | (미정의) | 전체 |
| 12 진료 목록 | status, keyword | (미정의) | 전체 |
| 15 예약 현황 | status, departmentId, doctorId, keyword | (미정의) | 전체 |
| 18 전체 예약 | 날짜범위, 상태, 진료과, 환자명 | date(단일), status, doctorId, page | 날짜범위, 진료과, 환자명 |
| 22 직원 목록 | ROLE, 재직여부, 이름 | role, isActive | 이름 검색 |
| 27 물품 목록 | category, stockStatus, keyword | category, shortage(Boolean) | keyword, stockStatus 형식 차이 |
| 30 규칙 목록 | 카테고리, 활성여부, 제목 | category, isActive | 제목 검색 |

### H-03. 예약 취소 권한 불일치 ✅ 조치 완료

> **조치:** API 명세서 v5, 화면 흐름 시퀀스 v5에서 예약 취소를 ADMIN 전용으로 통일

| 문서 | 취소 가능 ROLE |
|------|---------------|
| 스토리보드 (Slide 39) | **ADMIN만** |
| 화면 흐름 시퀀스 다이어그램 v4 | **STAFF + ADMIN** |

---

### H-04. ADMIN 교차 접근 권한 — 스토리보드에 미반영 ⚠️ 보류

> **보류 사유:** 정의서 v5에서 ADMIN 교차 접근 유지. 스토리보드에서의 미표기는 UI 표현의 제약으로 판단.

화면 기능 정의서 v4는 ADMIN이 하위 역할 화면(07~10, 12~13, 15~16)에 접근 가능하다고 정의하나, 스토리보드에는 반영되지 않음.

---

### H-05. 물품 담당자 역할 존재 여부 불일치 ⚠️ 보류

> **보류 사유:** ROLE_ADMIN 통합 유지. 물품 담당자는 ADMIN 역할 내에서 물품 관련 메뉴로 접근하는 것으로 통일. 별도 ROLE 추가 시 4-ROLE 체계 변경이 필요하여 현재 단계에서는 보류.

| 문서 | 물품 담당자 역할 |
|------|----------------|
| 스토리보드 | 별도 역할로 존재, 전용 대시보드·화면 흐름 |
| ERD v3 / 화면 정의서 v4 / 시퀀스 다이어그램 v4 | **존재하지 않음** — ROLE_ADMIN으로 통합 |

스토리보드 Slide 2에서 "물품 담당자"를 6가지 역할 중 하나로 표시하나, 다른 모든 문서에서는 ROLE_ADMIN에 통합됨.

---

### H-06. ITEM 카테고리 라벨 불일치 ⚠️ 보류

> **보류 사유:** ERD의 ENUM 코드명 유지, UI 표시명은 구현 시 스토리보드 한글명으로 매핑. 코드 변경 불필요.

| 스토리보드 | ERD v3 코드 | ERD v3 한글 |
|-----------|-------------|------------|
| 의료용품 | MEDICAL_SUPPLIES | 의료 소모품 |
| 의료기기 | MEDICAL_EQUIPMENT | 의료 장비 |
| 소모품 | GENERAL_SUPPLIES | 일반 소모품 |

"의료용품" ≠ "의료 소모품", "소모품" ≠ "일반 소모품" — 용어가 일치하지 않음.

---

### H-07. 관리자 전체 접수 목록 — 스토리보드에서 누락 ⚠️ 보류

> **보류 사유:** 정의서 v5에서 화면 19 (전체 접수 목록) 유지. 스토리보드 미반영은 PPTX 미수정 방침에 의한 것.

API 명세서 `GET /admin/reception/list` (#33) 및 화면 기능 정의서 화면 19에 정의된 전체 접수 목록 화면이 스토리보드에 없음.

---

### H-08. 관리자 대시보드 통계 — 직원 수 항목 불일치 ✅ 조치 완료

> **조치:** API 명세서 v5에서 관리자 대시보드 응답에 `staffCount` 필드 추가

| 항목 | 스토리보드 (화면 17) | API 명세서 v4 |
|------|---------------------|-------------|
| 통계 | 오늘예약, **총예약**, **직원수**, 재고부족, 진료과별 | todayReservationCount, totalReservationCount, departmentStats, itemShortageCount |

스토리보드에는 "직원 수" 통계가 있으나, API 명세서 응답에는 해당 필드 없음.

### M-06. 직원 등록 — specialty 필드 누락 ⚠️ 보류

> **보류 사유:** API 측 specialty 유지. 스토리보드에 미표시된 것은 UI 단축 표현으로 판단. 구현 시 포함.

API 명세서 v4의 `POST /admin/staff/create`에는 `specialty` 필드가 있으나, 스토리보드(화면 23)에는 해당 입력 필드가 없음.

### M-07. LLM 소견(opinion) — ERD에 전용 컬럼 없음 ⚠️ 보류

> **보류 사유:** `llm_response_raw` JSON에서 파싱하여 표시 가능. 별도 컬럼 추가는 과설계. 구현 시 Service 레이어에서 처리.

### M-08. PATIENT.updated_at — ERD에 없음 ✅ 조치 완료

> **조치:** ERD v4에 `PATIENT.updated_at` DATETIME NULL 추가

스토리보드 화면 08, 16에서 환자 정보를 수정하나, ERD PATIENT 테이블에 `updated_at` 컬럼이 없음.

### M-09. HOSPITAL_RULE.content 최대 길이 — ERD에 미명시 ✅ 조치 완료

> **조치:** ERD v4에서 `HOSPITAL_RULE.content` TEXT 컬럼에 "애플리케이션 레벨 3000자 제한" 주석 추가

스토리보드에서 규칙 내용 최대 3000자로 제한하나, ERD에는 TEXT 타입만 정의되어 있고 문자 수 제한이 없음.

### L-02. 상세 화면 URL에 쿼리 파라미터 미표기 ⚠️ 보류 (구현 시 해결)

스토리보드는 `GET /staff/reception/detail`과 같이 쿼리 파라미터 없이 URL만 표기.
정의서는 `GET /staff/reception/detail?reservationId={id}`로 파라미터를 명시.

해당 화면: 08, 13, 16, 24, 26, 29, 32 (총 7개)

---

### L-03. PRG 리다이렉트 대상 미표기 ⚠️ 보류 (구현 시 해결)

스토리보드에서 POST 후 리다이렉트 대상이 명시되지 않은 화면:
- 화면 09 (전화 예약 → ?), 화면 10 (방문 접수 → ?), 화면 13 (진료 기록 → ?)

시퀀스 다이어그램에서는 모두 `redirect:/목록` 으로 명확히 정의.

---

### L-04. AJAX 경로 축약 표기 ⚠️ 보류 (구현 시 해결)

스토리보드 화면 03에서 `GET /getDoctors`, `GET /getSlots`로 축약 표기.
API 명세서 전체 경로: `GET /reservation/getDoctors`, `GET /reservation/getSlots`.

---

### L-05. 예약 방식 선택 → LLM 실패 시 처리 미정의 ⚠️ 보류 (구현 시 해결)

정의서 v4에는 LLM 호출 실패 시 에러 toast + 3초 후 직접 선택(`/reservation/direct`) 자동 이동이 정의되어 있으나, 스토리보드에는 해당 오류 처리 흐름이 없음.

---

### L-06. 진료과 activate/deactivate API 분리 미반영 ⚠️ 보류 (구현 시 해결)

정의서 v4는 `POST /admin/department/activate`와 `POST /admin/department/deactivate`를 분리하나, 스토리보드는 단일 토글 형태로만 표현.

---

### L-07. 스토리보드 버전이 v3.0 ⚠️ 보류 (PPTX 미수정 방침)

스토리보드 표지에 "화면 정의서 v3.0" 기준으로 명시. v4.0에서 변경된 사항(진료과 API 분리, ITEM 카테고리 통일, 관리자 환자 관리 추가 등)이 반영되지 않음.
