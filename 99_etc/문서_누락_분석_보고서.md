| 컬럼명 | 타입 | 제약 | 설명 |

### 1-2. [심각] RESERVATION.source 컬럼 미정의

| 항목 | 내용 |
|------|------|
| **문제** | 예약 경로 구분 컬럼(`source`)이 ERD RESERVATION 테이블에 없음 |
| **근거 문서** | 화면 흐름 v3.0: `source=ONLINE`, `source=PHONE`, `source=WALKIN` 명시 |
| **근거 화면** | 화면 19 (전체 접수 목록): "예약 구분(ONLINE/PHONE/WALKIN)" 필터 존재 |
| **영향 범위** | 접수 목록 필터링, 통계 구분, 방문접수 vs 전화예약 vs 온라인예약 식별 불가 |
| **권장 조치** | RESERVATION 테이블에 `source VARCHAR(20) NOT NULL DEFAULT 'ONLINE'` 컬럼 추가 |

**ENUM 값:**

| 값 | 설명 | 생성 주체 |
|----|------|-----------|
| ONLINE | 비회원 온라인 예약 | 비회원 환자 |
| PHONE | 전화 예약 | ROLE_STAFF |
| WALKIN | 방문 접수 | ROLE_STAFF |

### 1-3. [중요] RESERVATION 테이블 — 추가 검토 필요 컬럼

| 컬럼 | 문제 | 근거 |
|------|------|------|
| `cancel_reason` | 취소 사유를 기록할 컬럼 없음 | 관리자/접수직원이 예약 취소 시 사유 기록 필요 가능성 |

## 2. PBL 최소 요구사항 미충족 사항

### 2-1. [심각] Redis 세션 저장소 — 전체 문서 미반영

| 항목 | 내용 |
|------|------|
| **PBL 요구** | "Redis를 이용한 세션 저장소 구성 (spring-session-data-redis 의존성 사용)" — 중요도: 최상 |
| **현재 상태** | 프로젝트 계획서, 아키텍처 정의서, API 명세서 모두 "세션 기반 인증"만 기술. Redis 언급 없음 |
| **영향** | PBL 필수 요구사항(7개 중 1개) 미충족 시 프로젝트 완료 불인정 위험 |
| **권장 조치** | 아키텍처 정의서에 Redis 설정 섹션 추가, `application.properties` 설정 예시 포함 |

**필요 설정 항목:**

```properties
# application.properties
spring.data.redis.host=localhost
spring.data.redis.port=6379
spring.session.store-type=redis
spring.session.redis.namespace=hospital:session
```

**필요 의존성:**

```gradle
implementation 'org.springframework.session:spring-session-data-redis'
implementation 'org.springframework.boot:spring-boot-starter-data-redis'
```

### 2-2. [심각] RESTful API HTTP 메서드 — GET/POST만 사용

| 항목 | 내용 |
|------|------|
| **PBL 요구** | "최소 3개 이상 RESTful 엔드포인트 (GET/POST/PUT/PATCH/DELETE 조합)" — 중요도: 최상 |
| **현재 상태** | API 명세서 v3.0 공통 규칙: "HTTP 메서드: GET/POST 두 종류만 사용" |
| **문제** | PUT, PATCH, DELETE 메서드가 전혀 사용되지 않음 |
| **영향** | PBL 필수 요구사항(7개 중 1개) 미충족 |
| **권장 조치** | AJAX 전용 엔드포인트 일부를 RESTful로 변경하거나, 별도 REST API 레이어 추가 |

**변경 대상 후보 (AJAX JSON 응답 엔드포인트):**

| 현재 | 변경 제안 | 메서드 |
|------|-----------|--------|
| `POST /admin/reservation/cancel` | `DELETE /api/reservations/{id}` | DELETE |
| `POST /admin/staff/update` | `PUT /api/staff/{id}` | PUT |
| `POST /nurse/patient/update` | `PATCH /api/patients/{id}` | PATCH |
| `POST /admin/item/delete` | `DELETE /api/items/{id}` | DELETE |
| `POST /admin/rule/delete` | `DELETE /api/rules/{id}` | DELETE |

> **참고:** SSR PRG 패턴의 POST 엔드포인트는 그대로 유지하면서, 별도 `/api/**` 경로로 RESTful 엔드포인트를 추가 제공하는 방식도 가능

### 2-3. [심각] 서버 사이드 유효성 검증 — @Valid + BindingResult 미문서화

| 항목 | 내용 |
|------|------|
| **PBL 요구** | "글로벌 예외 핸들러 (@ControllerAdvice) + 서버 사이드 유효성 검증 (@Valid + BindingResult)" — 중요도: 최상 |
| **현재 상태** | 아키텍처 정의서에 `GlobalExceptionHandler` 언급 있으나, `@Valid` + `BindingResult` 패턴 미기술 |
| **문제** | API 명세서의 POST 처리 패턴(PRG)에서 유효성 검증 흐름이 명시되지 않음 |
| **권장 조치** | API 명세서 또는 아키텍처 정의서에 DTO 유효성 검증 패턴 추가 |

**필요 문서화 내용:**

```java
// 예시: 예약 생성 DTO
public class ReservationCreateRequest {
    @NotBlank(message = "환자 이름은 필수입니다")
    private String patientName;

    @NotBlank @Pattern(regexp = "^\\d{2,3}-\\d{3,4}-\\d{4}$")
    private String patientPhone;

    @NotNull
    private Long departmentId;
    // ...
}

// Controller에서의 사용
@PostMapping("/reservation/create")
public String create(@Valid ReservationCreateRequest request,
                     BindingResult bindingResult,
                     HttpServletRequest httpRequest,
                     RedirectAttributes redirectAttributes) {
    if (bindingResult.hasErrors()) {
        httpRequest.setAttribute("errors", bindingResult.getAllErrors());
        httpRequest.setAttribute("inputData", request);
        return "reservation/direct";  // 폼 재렌더링
    }
    // ...
}
```

### 2-4. [중요] 페이징 구현 상세 미문서화

| 항목 | 내용 |
|------|------|
| **PBL 요구** | "Pageable 활용, Page\<T\> 또는 Slice\<T\> 반환 권장" |
| **현재 상태** | 화면 기능 정의서에 "페이징(20건)" 기능 명시되어 있으나, Spring Data JPA의 Pageable/Page\<T\> 사용 여부 미기술 |
| **권장 조치** | 아키텍처 정의서 또는 API 명세서에 페이징 구현 패턴 상세 추가 |

**필요 문서화 예시:**

```
GET /admin/reservation/list?page=0&size=20&sort=reservationDate,desc
→ Controller: @PageableDefault(size = 20, sort = "reservationDate", direction = DESC)
→ Service: Page<ReservationDto> 반환
→ Mustache: page.number, page.totalPages, page.totalElements 활용
```

## 5. 기능 설계 보완 필요 사항

### 5-1. CSRF 토큰 + AJAX 처리 방법 미문서화

| 항목 | 내용 |
|------|------|
| **문제** | AJAX 요청(6개 엔드포인트)에서 CSRF 토큰 전달 방법이 어느 문서에도 없음 |
| **영향** | PBL 주의사항에서도 "CSRF 토큰 문제 발생 가능" 경고 |
| **권장 조치** | 아키텍처 정의서에 CSRF 메타 태그 방식 또는 SecurityConfig에서 특정 경로 제외 방법 문서화 |

**해결 방안 예시:**

```html
<!-- Mustache 공통 레이아웃 head -->
<meta name="_csrf" content="{{_csrf.token}}"/>
<meta name="_csrf_header" content="{{_csrf.headerName}}"/>
```

```javascript
// AJAX 공통 설정
const csrfToken = document.querySelector('meta[name="_csrf"]').content;
const csrfHeader = document.querySelector('meta[name="_csrf_header"]').content;
fetch(url, {
    method: 'POST',
    headers: { [csrfHeader]: csrfToken, 'Content-Type': 'application/json' },
    body: JSON.stringify(data)
});
```
