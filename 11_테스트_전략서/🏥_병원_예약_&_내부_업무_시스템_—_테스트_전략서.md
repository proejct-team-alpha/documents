## 1. 테스트 목표 및 범위

### 1.1 테스트 목표

| 목표 | 설명 |
|------|------|
| 기능 정확성 | 예약, 접수, 진료, 관리자 CRUD 등 핵심 비즈니스 로직이 요구사항대로 동작하는지 검증 |
| 상태 전이 안정성 | 예약 상태 흐름(RESERVED -> RECEIVED -> IN_TREATMENT -> COMPLETED)이 정방향으로만 전이되는지 검증 |
| 보안 검증 | ROLE 기반 접근 제어(ADMIN, DOCTOR, NURSE, STAFF)가 올바르게 적용되는지 검증 |
| LLM 연동 안정성 | Claude API 호출 실패, 타임아웃, 비정상 응답 시 폴백이 정상 작동하는지 검증 |
| 데이터 무결성 | 중복 예약 방지, 유효성 검증(@Valid), 트랜잭션 원자성 보장 |
| 회귀 방지 | 코드 변경 시 기존 기능이 깨지지 않도록 자동화된 테스트 스위트 유지 |

### 1.2 테스트 범위

**In-Scope:**

- Service 레이어 비즈니스 로직 전체
- Repository 레이어 커스텀 쿼리 및 CRUD
- Controller 레이어 주요 엔드포인트 (PRG 패턴, ROLE 접근 제어, @Valid 검증)
- 예약 -> 접수 -> 진료 전체 흐름 통합 테스트
- LLM 연동 (Mock 기반)
- Spring Security 인증/인가

**Out-of-Scope:**

- 프론트엔드 UI 렌더링 (Mustache 템플릿 자체의 시각적 검증)
- 외부 Claude API 실제 호출 (비용 및 네트워크 의존성)
- 성능/부하 테스트 (4주 일정 내 범위 초과)
- E2E 브라우저 테스트 (Selenium 등)

## 3. 테스트 유형별 전략

### 3.1 단위 테스트 (Unit Test) -- Service 레이어

> **목적:** 비즈니스 로직의 정확성을 외부 의존성 없이 검증한다.
> **도구:** JUnit 5 + Mockito
> **대상:** 모든 Service 클래스

#### 핵심 테스트 항목

| 테스트 대상 | 검증 내용 | 예시 |
|-------------|-----------|------|
| **상태 전이 로직** | 정방향 전이만 허용, 역방향 차단 | RESERVED -> RECEIVED (성공), COMPLETED -> RESERVED (실패) |
| **중복 예약 방지** | 동일 의사 + 동일 날짜 + 동일 시간 슬롯 중복 차단 | `ReservationService.create()` 중복 시 예외 발생 |
| **LLM 폴백 로직** | API 실패 시 직접 선택 모드 전환 | `LlmService` 예외 발생 시 폴백 응답 반환 |
| **권한 검증** | 서비스 레이어에서 ROLE 확인 | STAFF만 접수 처리 가능 |
| **유효성 검증** | 비즈니스 규칙 검증 | 과거 날짜 예약 차단, 빈 증상 텍스트 차단 |

#### 코드 예시 -- 상태 전이 테스트

```java
@ExtendWith(MockitoExtension.class)
@DisplayName("ReservationService 단위 테스트")
class ReservationServiceTest {

    @Mock
    private ReservationRepository reservationRepository;

    @InjectMocks
    private ReservationService reservationService;

    @Nested
    @DisplayName("상태 전이 테스트")
    class StatusTransitionTest {

        @Test
        @DisplayName("RESERVED -> RECEIVED 전이 성공")
        void shouldTransitionFromReservedToReceived() {
            // given
            Reservation reservation = createReservation(ReservationStatus.RESERVED);
            when(reservationRepository.findById(1L))
                .thenReturn(Optional.of(reservation));

            // when
            reservationService.receive(1L);

            // then
            assertThat(reservation.getStatus()).isEqualTo(ReservationStatus.RECEIVED);
        }

        @Test
        @DisplayName("COMPLETED -> RESERVED 역방향 전이 차단")
        void shouldRejectReverseTransition() {
            // given
            Reservation reservation = createReservation(ReservationStatus.COMPLETED);
            when(reservationRepository.findById(1L))
                .thenReturn(Optional.of(reservation));

            // when & then
            assertThatThrownBy(() -> reservationService.receive(1L))
                .isInstanceOf(IllegalStateException.class)
                .hasMessageContaining("상태 전이 불가");
        }
    }

    @Nested
    @DisplayName("중복 예약 방지 테스트")
    class DuplicateReservationTest {

        @Test
        @DisplayName("동일 의사-날짜-시간 중복 예약 시 예외 발생")
        void shouldRejectDuplicateReservation() {
            // given
            when(reservationRepository.existsByDoctorIdAndDateAndTimeSlot(
                1L, LocalDate.of(2026, 3, 10), TimeSlot.AM_09))
                .thenReturn(true);

            // when & then
            assertThatThrownBy(() -> reservationService.create(duplicateRequest))
                .isInstanceOf(DuplicateReservationException.class);
        }
    }
}
```

### 3.2 Repository 테스트 (@DataJpaTest)

> **목적:** JPA 엔티티 매핑, 커스텀 쿼리, 페이징이 올바르게 동작하는지 검증한다.
> **도구:** JUnit 5 + @DataJpaTest + H2
> **대상:** 모든 Repository 인터페이스

#### 핵심 테스트 항목

| 테스트 대상 | 검증 내용 |
|-------------|-----------|
| **기본 CRUD** | save, findById, findAll, delete 동작 확인 |
| **페이징 쿼리** | `Pageable` 파라미터 적용 시 정확한 페이지 결과 반환 |
| **커스텀 쿼리** | `findByDoctorIdAndDate()`, `findByStatus()` 등 조건부 조회 |
| **UNIQUE 제약** | 중복 데이터 삽입 시 `DataIntegrityViolationException` 발생 |
| **연관관계 매핑** | N:1, 1:N 관계 로딩 및 영속성 검증 |

#### 코드 예시 -- Repository 테스트

```java
@DataJpaTest
@DisplayName("ReservationRepository 테스트")
class ReservationRepositoryTest {

    @Autowired
    private ReservationRepository reservationRepository;

    @Autowired
    private TestEntityManager em;

    @Test
    @DisplayName("의사ID + 날짜로 예약 목록 조회")
    void findByDoctorIdAndDate() {
        // given
        Doctor doctor = em.persist(createDoctor("김철수"));
        em.persist(createReservation(doctor, LocalDate.of(2026, 3, 10)));
        em.persist(createReservation(doctor, LocalDate.of(2026, 3, 10)));
        em.persist(createReservation(doctor, LocalDate.of(2026, 3, 11)));

        // when
        List<Reservation> result = reservationRepository
            .findByDoctorIdAndDate(doctor.getId(), LocalDate.of(2026, 3, 10));

        // then
        assertThat(result).hasSize(2);
    }

    @Test
    @DisplayName("페이징 조회 정상 동작")
    void findAllWithPaging() {
        // given
        for (int i = 0; i < 25; i++) {
            em.persist(createReservation());
        }

        // when
        Page<Reservation> page = reservationRepository
            .findAll(PageRequest.of(0, 10, Sort.by("createdAt").descending()));

        // then
        assertThat(page.getContent()).hasSize(10);
        assertThat(page.getTotalElements()).isEqualTo(25);
        assertThat(page.getTotalPages()).isEqualTo(3);
    }

    @Test
    @DisplayName("동일 의사-날짜-시간 중복 저장 시 예외")
    void shouldRejectDuplicateSlot() {
        // given
        Doctor doctor = em.persist(createDoctor("김철수"));
        em.persist(createReservation(doctor, LocalDate.of(2026, 3, 10), TimeSlot.AM_09));

        // when & then
        assertThatThrownBy(() -> {
            em.persist(createReservation(doctor, LocalDate.of(2026, 3, 10), TimeSlot.AM_09));
            em.flush();
        }).isInstanceOf(Exception.class);
    }
}
```

### 3.3 Controller 테스트 (MockMvc)

> **목적:** HTTP 요청/응답, PRG 패턴, ROLE 접근 제어, @Valid 검증 실패 처리를 검증한다.
> **도구:** JUnit 5 + MockMvc + @WebMvcTest + Spring Security Test
> **대상:** 모든 Controller 클래스

#### 핵심 테스트 항목

| 테스트 대상 | 검증 내용 |
|-------------|-----------|
| **PRG 패턴** | POST 처리 후 redirect 응답 (302), GET으로 결과 화면 반환 |
| **ROLE 접근 제어** | 권한 없는 ROLE이 접근 시 403 또는 로그인 페이지 리다이렉트 |
| **@Valid 검증 실패** | 유효하지 않은 입력 시 BindingResult 에러와 함께 폼 화면 재표시 |
| **CSRF 토큰** | POST 요청 시 CSRF 토큰 포함 여부 확인 |
| **정상 경로** | 200 OK 응답, 올바른 뷰 이름 반환, 모델 속성 확인 |

#### 코드 예시 -- Controller 테스트

```java
@WebMvcTest(ReservationController.class)
@DisplayName("ReservationController 테스트")
class ReservationControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private ReservationService reservationService;

    @MockBean
    private DepartmentService departmentService;

    @Test
    @DisplayName("예약 생성 성공 시 PRG 패턴으로 리다이렉트")
    void shouldRedirectAfterCreate() throws Exception {
        // given
        when(reservationService.create(any())).thenReturn(1L);

        // when & then
        mockMvc.perform(post("/reservation/create")
                .param("patientName", "홍길동")
                .param("phoneNumber", "010-1234-5678")
                .param("departmentId", "1")
                .param("doctorId", "1")
                .param("date", "2026-03-10")
                .param("timeSlot", "AM_09")
                .with(csrf()))
            .andExpect(status().is3xxRedirection())
            .andExpect(redirectedUrlPattern("/reservation/complete*"));
    }

    @Test
    @DisplayName("@Valid 검증 실패 시 폼 화면 재표시")
    void shouldReturnFormOnValidationFailure() throws Exception {
        mockMvc.perform(post("/reservation/create")
                .param("patientName", "")  // 빈 이름 -- @NotBlank 위반
                .param("phoneNumber", "")  // 빈 전화번호
                .with(csrf()))
            .andExpect(status().isOk())
            .andExpect(view().name("reservation/form"))
            .andExpect(model().attributeHasFieldErrors(
                "reservationForm", "patientName", "phoneNumber"));
    }

    @Test
    @DisplayName("ROLE_STAFF 권한으로 접수 처리 접근 성공")
    @WithMockUser(roles = "STAFF")
    void shouldAllowStaffToAccessReception() throws Exception {
        mockMvc.perform(get("/staff/reception/list"))
            .andExpect(status().isOk());
    }

    @Test
    @DisplayName("ROLE_DOCTOR 권한으로 접수 처리 접근 차단")
    @WithMockUser(roles = "DOCTOR")
    void shouldDenyDoctorFromAccessingReception() throws Exception {
        mockMvc.perform(get("/staff/reception/list"))
            .andExpect(status().isForbidden());
    }
}
```

### 3.4 통합 테스트 (@SpringBootTest)

> **목적:** 예약 -> 접수 -> 진료 전체 흐름이 Application Context 전체에서 정상 동작하는지 검증한다.
> **도구:** JUnit 5 + @SpringBootTest + @AutoConfigureMockMvc
> **시점:** W4 통합 테스트 기간

#### 핵심 시나리오

| 시나리오 | 흐름 |
|----------|------|
| **전체 예약 흐름** | 예약 생성(비회원) -> 접수(STAFF) -> 진료(DOCTOR) -> 완료 |
| **LLM 추천 예약** | 증상 입력 -> LLM 추천 -> 추천 기반 예약 -> 접수 -> 진료 |
| **직접 선택 폴백** | 증상 입력 -> LLM 실패 -> 직접 선택 화면 전환 -> 예약 |
| **관리자 CRUD** | 규칙 등록 -> 챗봇에서 해당 규칙 질의 -> 정상 응답 확인 |

#### 코드 예시 -- 통합 테스트

```java
@SpringBootTest
@AutoConfigureMockMvc
@Transactional
@DisplayName("예약 -> 접수 -> 진료 전체 흐름 통합 테스트")
class ReservationFlowIntegrationTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private ReservationRepository reservationRepository;

    @Test
    @DisplayName("예약 생성 -> 접수 -> 진료 완료 전체 흐름")
    void fullReservationFlow() throws Exception {
        // 1. 예약 생성 (비회원)
        mockMvc.perform(post("/reservation/create")
                .param("patientName", "홍길동")
                .param("phoneNumber", "010-1234-5678")
                .param("departmentId", "1")
                .param("doctorId", "1")
                .param("date", "2026-03-10")
                .param("timeSlot", "AM_09")
                .with(csrf()))
            .andExpect(status().is3xxRedirection());

        Reservation reservation = reservationRepository.findAll().get(0);
        assertThat(reservation.getStatus()).isEqualTo(ReservationStatus.RESERVED);

        // 2. 접수 처리 (STAFF)
        mockMvc.perform(post("/staff/reception/receive/" + reservation.getId())
                .with(user("staff1").roles("STAFF"))
                .with(csrf()))
            .andExpect(status().is3xxRedirection());

        em.flush();
        em.clear();

        reservation = reservationRepository.findById(reservation.getId()).orElseThrow();
        assertThat(reservation.getStatus()).isEqualTo(ReservationStatus.RECEIVED);

        // 3. 진료 완료 (DOCTOR)
        mockMvc.perform(post("/doctor/treatment/complete/" + reservation.getId())
                .param("diagnosis", "급성 장염")
                .param("prescription", "소화제 처방")
                .with(user("doctor1").roles("DOCTOR"))
                .with(csrf()))
            .andExpect(status().is3xxRedirection());

        em.flush();
        em.clear();

        reservation = reservationRepository.findById(reservation.getId()).orElseThrow();
        assertThat(reservation.getStatus()).isEqualTo(ReservationStatus.COMPLETED);
    }
}
```

## 4. 테스트 커버리지 목표

### 4.1 레이어별 커버리지 기준

| 레이어 | 목표 커버리지 | 우선순위 | 비고 |
|--------|--------------|----------|------|
| **Service** | **80% 이상** | 최우선 | 비즈니스 로직 핵심 -- 상태 전이, 중복 방지, LLM 폴백 |
| **Repository** | **70% 이상** | 높음 | 커스텀 쿼리, 페이징, UNIQUE 제약 |
| **Controller** | **주요 경로 커버** | 중간 | PRG 패턴, ROLE 접근 제어, @Valid 검증 실패 |
| **Config / Security** | 주요 설정 검증 | 중간 | URL별 ROLE 매핑, CSRF 설정 |
| **DTO / Entity** | 제외 | 낮음 | 단순 getter/setter는 테스트 불필요 |

### 4.2 핵심 클래스별 커버리지 목표

| 클래스 | 목표 | 이유 |
|--------|------|------|
| `ReservationService` | 90%+ | 예약 생성, 상태 전이, 중복 방지 등 핵심 로직 집중 |
| `LlmService` | 85%+ | API 호출, 응답 파싱, 폴백 처리 모두 검증 필요 |
| `SlotService` | 85%+ | 시간 슬롯 가용 여부 계산 로직 |
| `SecurityConfig` | URL 패턴 전체 | 모든 ROLE URL 접근 제어 검증 |
| `ReceptionService` | 80%+ | 접수 상태 전이 로직 |
| `TreatmentService` | 80%+ | 진료 기록 저장 및 완료 처리 |

## 6. LLM 관련 테스트 전략

### 6.1 테스트 원칙

> Claude API는 **절대 실제 호출하지 않는다.**
> 모든 LLM 테스트는 Mockito를 사용하여 `LlmService` 내부의 HTTP 클라이언트를 Mock 처리한다.

### 6.2 Mock 처리 패턴

```java
@ExtendWith(MockitoExtension.class)
@DisplayName("LlmService 테스트")
class LlmServiceTest {

    @Mock
    private RestTemplate restTemplate;  // 또는 WebClient

    @Mock
    private HospitalRuleRepository hospitalRuleRepository;

    @InjectMocks
    private LlmService llmService;

    @Test
    @DisplayName("증상 분석 정상 응답 파싱")
    void shouldParseSymptomAnalysisResponse() {
        // given
        String mockResponse = """
            {
                "recommended_department": "외과",
                "recommended_doctor": "김철수",
                "recommended_time": "오전",
                "summary": "복통 증상으로 외과 진료가 적합합니다.",
                "disclaimer": "본 안내는 AI 참고용이며 의학적 진단이 아닙니다."
            }
            """;

        when(restTemplate.postForObject(anyString(), any(), eq(String.class)))
            .thenReturn(mockResponse);

        // when
        LlmRecommendation result = llmService.analyzeSymptom("배가 아프고 미열이 있습니다");

        // then
        assertThat(result.getRecommendedDepartment()).isEqualTo("외과");
        assertThat(result.getRecommendedDoctor()).isEqualTo("김철수");
        assertThat(result.getDisclaimer()).isNotBlank();
    }
}
```

### 6.3 LLM 테스트 시나리오 목록

| 시나리오 | 검증 내용 | Mock 설정 |
|----------|-----------|-----------|
| **정상 응답** | JSON 파싱 성공, 추천 결과 반환 | `when().thenReturn(정상 JSON)` |
| **타임아웃** | 5초 초과 시 폴백 응답 반환 | `when().thenThrow(new SocketTimeoutException())` |
| **API 오류 (5xx)** | Claude API 서버 오류 시 폴백 | `when().thenThrow(new HttpServerErrorException(...))` |
| **API 키 미설정** | 키 없이 호출 시 적절한 예외 | `when().thenThrow(new HttpClientErrorException(UNAUTHORIZED))` |
| **비정상 JSON** | 파싱 불가 응답 시 폴백 | `when().thenReturn("invalid json")` |
| **빈 응답** | null 또는 빈 문자열 응답 시 폴백 | `when().thenReturn("")` |
| **폴백 동작** | 모든 실패 케이스에서 "직접 선택" 안내 반환 | 폴백 메서드 반환값 확인 |
| **챗봇 정상** | 병원 규칙 기반 답변 반환 | `when().thenReturn(규칙 답변 JSON)` |
| **챗봇 규칙 없음** | 등록된 규칙이 없을 때 안내 메시지 | 빈 규칙 목록 Mock |

### 6.4 폴백 테스트 상세

```java
@Test
@DisplayName("Claude API 타임아웃 시 직접 선택 폴백 응답 반환")
void shouldFallbackOnTimeout() {
    // given
    when(restTemplate.postForObject(anyString(), any(), eq(String.class)))
        .thenThrow(new ResourceAccessException("Read timed out"));

    // when
    LlmRecommendation result = llmService.analyzeSymptom("두통이 있습니다");

    // then
    assertThat(result.isFallback()).isTrue();
    assertThat(result.getSummary()).contains("직접 선택");
}

@Test
@DisplayName("비정상 JSON 응답 시 폴백 처리")
void shouldFallbackOnInvalidJson() {
    // given
    when(restTemplate.postForObject(anyString(), any(), eq(String.class)))
        .thenReturn("이것은 JSON이 아닙니다");

    // when
    LlmRecommendation result = llmService.analyzeSymptom("기침이 심합니다");

    // then
    assertThat(result.isFallback()).isTrue();
}
```

### 8.1 GitHub Actions 워크플로우 (권장)

```yaml
# .github/workflows/test.yml
name: Test

on:
  pull_request:
    branches: [develop, main]
  push:
    branches: [develop]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      mysql:
        image: mysql:8.0
        env:
          MYSQL_ROOT_PASSWORD: testpassword
          MYSQL_DATABASE: hospital_test
        ports:
          - 3306:3306
        options: >-
          --health-cmd="mysqladmin ping"
          --health-interval=10s
          --health-timeout=5s
          --health-retries=5

      redis:
        image: redis:7.0
        ports:
          - 6379:6379

    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Grant execute permission for gradlew
        run: chmod +x gradlew

      - name: Run tests
        run: ./gradlew test
        env:
          SPRING_DATASOURCE_URL: jdbc:mysql://localhost:3306/hospital_test
          SPRING_DATASOURCE_USERNAME: root
          SPRING_DATASOURCE_PASSWORD: testpassword
          SPRING_DATA_REDIS_HOST: localhost
          SPRING_DATA_REDIS_PORT: 6379
          CLAUDE_API_KEY: test-mock-key

      - name: Generate JaCoCo Report
        run: ./gradlew jacocoTestReport

      - name: Upload Test Report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-report
          path: build/reports/tests/test/

      - name: Upload Coverage Report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: build/reports/jacoco/test/html/
```

## 9. 테스트 작성 규칙

### 9.1 테스트 코드 컨벤션

| 규칙 | 설명 | 예시 |
|------|------|------|
| **@DisplayName 필수** | 한글로 테스트 의도를 명확히 기술 | `@DisplayName("동일 시간 중복 예약 시 예외 발생")` |
| **Given-When-Then 패턴** | 주석으로 구조 구분 | `// given`, `// when`, `// then` |
| **@Nested 활용** | 테스트 대상별 그룹핑 | `@Nested class StatusTransitionTest` |
| **한 테스트 한 검증** | 하나의 @Test에서 하나의 시나리오만 검증 | 상태 전이 성공과 실패를 별도 메서드로 분리 |
| **테스트 독립성** | 테스트 간 상태 공유 금지, 실행 순서 의존 금지 | `@Transactional` 또는 `@BeforeEach` 초기화 |
| **매직 넘버 금지** | 상수 또는 헬퍼 메서드 사용 | `createReservation(DOCTOR_KIM, MARCH_10, AM_09)` |
