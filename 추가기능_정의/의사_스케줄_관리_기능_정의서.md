# 의사 스케줄 관리 기능 정의서

> **문서 유형:** 추가기능 정의 (스토리보드 미포함, 기존 문서에 존재)
> **작성일:** 2026년
> **연관 문서:** ERD v4, API 명세서 v5, 화면 기능 정의서 v5, 화면 흐름 시퀀스 다이어그램 v5

---

## 1. 개요

### 1.1 배경

의사 스케줄(진료 가능 요일)은 ERD, API, 화면 정의서 등 기존 문서에 `DOCTOR.available_days`로 정의되어 있으나, **스토리보드에는 별도 화면으로 명시되지 않았다.** 현재는 직원 등록·수정 화면(화면 23, 24)에서 DOCTOR 역할 선택 시 진료 요일을 함께 입력·수정하는 방식으로 구현되어 있다.

본 문서는 의사 스케줄 관리 기능을 명확히 정의하고, 예약 흐름·관리자 CRUD·API 연동을 정리한다.

### 1.2 기능 요약

| 항목 | 내용 |
|------|------|
| **데이터 소스** | `DOCTOR.available_days` (VARCHAR, 예: `MON,WED,FRI`) |
| **예약 시 조회** | 사용자 지정 예약(직접 선택), AI 추천 예약 모두에서 의사별 진료 가능 요일 기반 필터링 |
| **관리자 CRUD** | 관리자 화면에서 CRUD 가능 — 직원·의사 관리(화면 23, 24)에서 진료 요일 입력·수정 |

## 2. 데이터 구조

### 2.1 ERD 연관 (DOCTOR 테이블)

| 컬럼명 | 타입 | 제약 | 설명 |
|--------|------|------|------|
| `available_days` | VARCHAR(20) | NULL | 진료 가능 요일 (예: `MON,WED,FRI`) |

**설계 포인트**

- 쉼표 구분 문자열로 저장한다. (예: `"MON,TUE,THU"`)
- `NULL`인 경우 모든 평일(MON~FRI) 진료 가능으로 간주한다.
- 요일 코드: `MON`, `TUE`, `WED`, `THU`, `FRI`, `SAT`, `SUN`

### 2.2 UI ↔ DB 형식 변환

| 구분 | 형식 | 예시 |
|------|------|------|
| **DB 저장** | 영문 콤마 구분 | `MON,WED,FRI` |
| **화면 표시** | 한글 체크박스 | ☑월 ☐화 ☑수 ☐목 ☑금 ☐토 ☐일 |

## 3. 예약 시 사용처

### 3.1 사용자 지정 예약 (직접 선택)

| 단계 | 설명 | API 연동 |
|------|------|----------|
| 진료과 선택 | 사용자가 진료과 선택 | — |
| 날짜 선택 | 사용자가 예약 날짜 선택 (yyyy-MM-dd) | — |
| 의사 목록 조회 | **선택된 날짜의 요일**에 진료 가능한 의사만 필터링하여 표시 | `GET /reservation/getDoctors?departmentId={id}&date={yyyy-MM-dd}` |
| 시간 슬롯 조회 | 의사·날짜 기준 예약 가능 슬롯 조회 | `GET /reservation/getSlots?doctorId={id}&date={yyyy-MM-dd}` |
| 예약 생성 | SlotService에서 `available_days` 기반 요일 검증 수행 | `POST /reservation/create` |

**검증 로직**

- 예약 생성 시 `reservationDate`의 요일이 해당 의사의 `available_days`에 포함되지 않으면 `DOCTOR_NOT_AVAILABLE` 에러 반환.
- `available_days`가 NULL인 경우 모든 평일(MON~FRI) 진료 가능으로 간주.

### 3.2 AI 추천 예약

| 단계 | 설명 | API 연동 |
|------|------|----------|
| 증상 입력 | 사용자가 증상 텍스트 입력 | `POST /llm/symptom/analyze` |
| LLM 추천 | Claude API가 진료과·의사·시간대 추천 | — |
| **프롬프트 주입** | `DOCTOR.specialty`, `DOCTOR.available_days`를 시스템 프롬프트에 주입하여 LLM이 진료 가능한 의사만 추천하도록 유도 | — |
| 추천 결과 표시 | 추천된 진료과·의사·시간 표시 | — |
| 예약 확정 | "이 추천으로 예약하기" 클릭 시 예약 폼으로 자동 입력 후 `POST /reservation/create` | `POST /reservation/create` |

**검증 로직**

- 예약 생성 시 동일하게 `available_days` 기반으로 SlotService에서 요일 검증 수행.

## 4. 관리자 CRUD

### 4.1 현재 구현 방식 (직원 관리 통합)

의사 스케줄은 **직원 등록·수정 화면**에서 DOCTOR 역할인 경우에만 관리된다.

| 화면 | URL | 기능 |
|------|-----|------|
| **화면 23 — 직원 등록** | `GET /admin/staff/new` | ROLE=DOCTOR 선택 시 진료과·전문 분야·**진료 요일** 입력 폼 표시 |
| **화면 24 — 직원 상세·수정** | `GET /admin/staff/detail?staffId={id}` | DOCTOR인 경우 진료과·전문 분야·**진료 요일** 수정 가능 |

**API 연동**

| API | 설명 |
|-----|------|
| `POST /admin/staff/create` | 직원 등록 시 DOCTOR인 경우 Staff + Doctor(available_days 포함) 저장 |
| `POST /admin/staff/update` | 직원 수정 시 DOCTOR인 경우 available_days 수정 |
| `PUT /api/staff/{id}` | RESTful API — 직원 수정 (availableDays 포함) |

### 4.2 관리자 화면 CRUD 요구사항

- **Create:** 직원 등록 시 DOCTOR 선택 → 진료 요일 입력 폼 노출 → `POST /admin/staff/create`
- **Read:** 직원 목록·상세에서 의사별 진료 요일 표시
- **Update:** 직원 상세·수정에서 진료 요일 체크박스 수정 → `POST /admin/staff/update`
- **Delete:** 진료 요일 삭제는 "전체 해제" 개념 — 모든 요일 체크 해제 시 `available_days = NULL` (평일 전체 진료 가능)

## 5. API 연동 요약

| API | 용도 | available_days 관련 |
|-----|------|---------------------|
| `GET /reservation/getDoctors` | 직접 선택 예약 — 의사 목록 조회 | `date` 파라미터 전달 시 available_days 기반 필터링 |
| `GET /reservation/getSlots` | 직접 선택 예약 — 시간 슬롯 조회 | 의사·날짜 조합 시 SlotService 내부에서 available_days 검증 |
| `POST /reservation/create` | 예약 생성 — 사용자 지정 | SlotService에서 available_days 검증 |
| `POST /staff/reservation/create` | 전화 예약 — STAFF 대리 예약 | SlotService에서 available_days 검증 |
| `POST /staff/walkin/create` | 방문 접수 | SlotService에서 available_days 검증 |
| `POST /staff/reservation/create` | 전화 예약 | SlotService에서 available_days 검증 |
| `POST /admin/staff/create` | 직원 등록 시 DOCTOR | available_days 저장 |
| `POST /admin/staff/update` | 직원 수정 시 DOCTOR | available_days 수정 |
| `PUT /api/staff/{id}` | RESTful 직원 수정 | availableDays 필드 |

## 6. 에러 코드

| 에러 코드 | HTTP | 설명 |
|-----------|------|------|
| `DOCTOR_NOT_AVAILABLE` | 400 | 예약 요청 시 해당 날짜의 요일이 의사의 `available_days`에 포함되지 않음 |

## 7. 향후 확장 고려 (선택)

스토리보드에 전용 화면이 추가될 경우, 아래와 같은 **의사 스케줄 전용 관리 화면**을 고려할 수 있다.

| 화면 | URL | 설명 |
|------|-----|------|
| 의사 스케줄 목록 | `GET /admin/doctor-schedule/list` | 의사별 진료 요일 일괄 조회·수정 |
| 의사 스케줄 수정 | `GET /admin/doctor-schedule/edit?doctorId={id}` | 특정 의사의 진료 요일만 단독 수정 |

> **현재 권장:** 직원 관리(화면 23, 24)에서 진료 요일을 함께 관리하는 방식으로 충분. 전용 화면은 의사 수가 많을 때 또는 스케줄 관리 빈도가 높을 때 검토.
